# AI Request Instructions

This file helps make future AI requests for the Quickspaces VS Code extension clear, consistent, and easy to execute.

## Purpose

Use this file when asking for code, documentation, or configuration changes in this repository. It helps define the goal, scope, constraints, and expected output.

## Recommended request structure

1. Goal
   - Clearly describe what you want to achieve.
   - Example: "Add a command to `src/extension.ts` that opens the current workspace in a new window."

2. Files or areas to modify
   - List the files or project areas involved.
   - Example: `src/extension.ts`, `package.json`, `README.md`

3. Constraints or preferences
   - Specify constraints such as minimal changes, dependency limits, or style restrictions.
   - Example: "Do not add new dependencies", "Keep the change contained to `src/` only", "Update tests if necessary."

4. Expected outcome
   - Describe the desired result, including behavior and deliverables.
   - Example: "Implement the feature, add command registration, and update docs." 

5. Deliverables
   - Specify whether you want code only, tests, docs, or a combination.
   - Example: "Add implementation and a test", "Add docs only", "Review current design and suggest improvements."

## Current repo structure

- `src/` – extension implementation in TypeScript.
- `resources/` – icon and resource files.
- `test/` – extension unit/integration test code.
- `package.json` – scripts, dependencies, and extension metadata.
- `tsconfig.json` – TypeScript compiler settings.
- `README.md` – user-facing extension documentation.

## Useful repository commands

- `yarn run compile` - compile TypeScript, run lint, and build extension artifacts.
- `yarn run package` - build extension for publishing.
- `yarn run watch` - run TypeScript and esbuild watches in parallel.
- `yarn run lint` - run ESLint on `src/`.
- `yarn run check-types` - type-check the project.
- `yarn run test` - run extension tests.
- `yarn run test:headless` - run tests in headless mode.

## Code quality expectations

- Keep changes minimal and targeted unless the request explicitly asks for broader refactoring.
- Follow existing TypeScript and VS Code extension patterns.
- Prefer small, self-contained commits for each feature or fix.
- Use existing commands and views rather than adding unnecessary UI complexity.

## Request checklist for AI changes

- [ ] Did you identify the exact files to modify?
- [ ] Did you state the feature or bug fix clearly?
- [ ] Did you include constraints (no new deps, keep change small, etc.)?
- [ ] Did you specify whether tests or docs should be included?
- [ ] Did you mention if the user wants a review before code changes?

## Example requests

- "Add a new command to `src/extension.ts` to refresh the quickspaces control plane and register it in `package.json`. Keep the change minimal."
- "Create a helper module for API integration, update the extension to use it, and add a short README section describing the new behavior."
- "Fix the `quickspaces.refresh` command so it updates the view correctly, and add a regression test in `src/test/`."

## Notes for reviewers

- If the request touches extension activation or command registration, include `package.json` and `src/extension.ts`.
- If the request touches UI or view items, mention `views`, `viewsWelcome`, or command contributions explicitly.
- If you want a design suggestion before implementation, say: "Review the current architecture and recommend the best place to add this feature."
