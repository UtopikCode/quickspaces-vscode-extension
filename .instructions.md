# AI Code Generation Guidelines – QuickSpaces VS Code Extension (v1)

This file provides **authoritative instructions** for AI assistants generating or modifying code in the **QuickSpaces VS Code Extension**.

The extension is a **pure UX and control surface**. It:
- Collects explicit user intent
- Displays workspace state and diagnostics
- Calls the QuickSpaces Control Plane API

It **does not execute infrastructure**, **does not manage lifecycle autonomously**, and **does not act as a controller**.

---

## 🚨 Core Architectural Invariants (DO NOT VIOLATE)

These rules override all other guidance:

- **The VS Code extension is never a controller**
  - No background polling loops
  - No automatic retries
  - No autonomous reconciliation
- **All execution is explicit**
  - Only triggered by direct user actions (clicks, commands)
- **The extension never calls the Runner**
  - All lifecycle actions go through the Control Plane API
- **No infrastructure logic**
  - No cloud SDKs
  - No VM, container, or SSH logic
- **No credentials persistence beyond session scope**

If generated code violates these rules, it is incorrect.

---

## Extension Responsibilities (v1)

The extension **MUST**:

- Authenticate the user via VS Code auth providers (GitHub, etc.)
- Initiate workspace lifecycle commands explicitly:
  - create
  - start
  - stop
  - delete
- Display:
  - desired_state
  - actual_state
  - lifecycle progress
  - error messages from the control plane
- Offer **explicit Retry actions** on failure

The extension **MUST NOT**:

- Retry automatically
- Poll workspace state in a loop
- Infer intent from state drift
- Execute infrastructure
- Store long‑lived tokens

---

## Communication Model

- The extension communicates **only** with the **QuickSpaces Control Plane API**
- All calls are **HTTP(S) request/response**
- Authentication uses **VS Code authentication APIs**
- OAuth tokens are:
  - Acquired on demand
  - Scoped to the current session
  - Never stored persistently by the extension

---

## Workspace Creation UX Rules

When creating a workspace, the extension:

1. Prompts the user to select:
   - Repository provider
   - Repository
   - Branch
   - Execution profile
2. Sends a single create request to the control plane
3. Performs no execution locally

Notes:
- Execution profiles map to:
  - adapter_type
  - runtime_config
- VM hosts, cloud accounts, regions, or runners are **never shown** to the user

---

## Lifecycle Actions

Each lifecycle action:

- Is triggered by an explicit UI command
- Sends exactly **one** API request
- Displays progress while the request is in flight
- Ends immediately when the response returns

There are **no background jobs** and **no watchers**.

---

## Error Handling & Retries

- Errors returned by the control plane must be surfaced verbatim
- The UI may present a **Retry button**
- Retrying equals a **new explicit lifecycle request**
- Never retry automatically

---

## Non‑Goals (v1)

The extension is **explicitly not responsible for**:

- Reconciling desired vs actual state
- Monitoring execution progress independently
- Acting when the UI is closed
- Enforcing infrastructure policy
- Running long‑lived services

---

## Technology & Style Guidelines

- Language: TypeScript
- Target: Latest stable VS Code extension API
- Keep extension activation lightweight
- Dispose all resources correctly
- Avoid global mutable state

---

## Final Rule to Remember

> **The extension asks. The control plane decides. The runner executes.**

If code violates this rule, it breaks QuickSpaces v1.
