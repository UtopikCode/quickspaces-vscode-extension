# AI Code Generation Guidelines – QuickSpaces VS Code Extension (v1)

This file provides **authoritative instructions** for AI assistants generating or modifying code in the **QuickSpaces VS Code Extension**.

The extension is a **pure UX and control surface**. It:
- Collects explicit user intent
- Displays workspace state and diagnostics
- Calls the QuickSpaces Control Plane API

It **does not execute infrastructure**, **does not manage lifecycle autonomously**, and **does not act as a controller**.

---

## 🚨 Core Architectural Invariants (DO NOT VIOLATE)

These rules override all other guidance:

- **The VS Code extension is never a controller**
  - No background polling loops
  - No automatic retries
  - No autonomous reconciliation
- **All execution is explicit**
  - Only triggered by direct user actions (clicks, commands)
- **The extension never calls the Runner**
  - All lifecycle actions go through the Control Plane API
- **No infrastructure logic**
  - No cloud SDKs
  - No VM, container, or SSH logic
- **No credentials persistence beyond session scope**

If generated code violates these rules, it is incorrect.

---

## Repository Role

This repository is a **VS Code extension / control surface**.

It is responsible for:
- Capturing explicit user intent
- Rendering workspace and control plane state
- Submitting explicit lifecycle actions to the Control Plane
- Managing short-lived authentication via VS Code auth APIs
- Providing a lightweight, responsive extension UI

It is not responsible for:
- Deciding when to execute work
- Executing infrastructure or runner behavior
- Reconciling desired vs actual state
- Holding or using infrastructure credentials
- Operating when the user closes VS Code

---

## Dependency Direction

This extension depends on:
- The QuickSpaces Control Plane API
- VS Code extension APIs and runtime libraries
- Node.js standard libraries for local behavior

This extension must not depend on:
- Runner or adapter implementations
- Infrastructure plumbing libraries or cloud provider SDKs
- Control Plane internal implementation details

Direction of responsibility:
- UI -> Control Plane
- Control Plane -> Runner
- Runner -> Execution Adapter

---

## Extension Responsibilities (v1)

The extension **MUST**:
- Authenticate users via VS Code authentication APIs
- Send explicit create, start, stop, delete, and refresh requests
- Display desired_state, actual_state, progress, and control plane errors
- Surface explicit Retry actions for user-driven retries
- Keep activation lightweight and dispose resources properly

The extension **MUST NOT**:
- Retry automatically
- Poll workspace state continuously
- Infer intent from state drift
- Manage infrastructure lifecycle locally
- Persist long-lived tokens or credentials

---

## Development Hygiene

- Language: TypeScript
- Follow existing ESLint rules and the current code style
- Prefer `async` / `await` over Promise chains when implementing async logic
- Avoid `any` except for explicit boundary handling
- Catch and propagate errors at the UI boundary
- Do not swallow errors silently
- Do not log or expose secrets
- Document exported helpers, commands, and API wrappers using TSDoc-style comments
- Keep command handlers and view providers small and focused
- Avoid global mutable state

Tests in this repository live under `/src/test` and should follow existing patterns.

Common commands:
- `npm run lint`
- `npm run check-types`
- `npm run compile`
- `npm run test`
- `npm run watch`

AI implementation checklist:
- Confirm this repo is the VS Code extension, not Control Plane or Runner
- Preserve `.github/workflows` and existing CI behavior
- Do not add background polling or reconciliation behavior
- Keep authentication short-lived and session-scoped
- Add tests for new behavior
- Keep UI activation lightweight and avoid retaining resources after disposal

---

## Final Rule to Remember

> **The extension asks. The control plane decides. The runner executes.**

If code violates this rule, it breaks QuickSpaces v1.
